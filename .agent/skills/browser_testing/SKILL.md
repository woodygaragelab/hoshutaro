# Browser UI Testing Skill

このスキルは、UIコンポーネントの改修後や新機能の追加後に、`browser_subagent` を使って画面の描画や挙動を検証・テストする際の標準化手順を定義します。AI（Antigravity）自身がこれを参照することでテストの品質と確実性を高めます。

## 1. 検証前のアプローチ

- UIやフロントエンドの実装（HTML, CSS, Reactなど）を変更した場合、**必ず一度はこのスキルに基づいてブラウザ検証を実施**すること。
- 検証前に現在の開発サーバーのポート番号（例: `5174`）とホストをローカル環境（`http://localhost:5174` 等）で動いているか確認する。

### 🚨 重要: データ初期ロードのボトルネック回避
多くの場合、サブエージェントが開くブラウザは初期状態のクリーンな環境であるため、**「データがありません」という初期画面でスタックアウトし、グリッド自体のテストができなくなります**。
これを回避するため、サブエージェントにテストを依頼する際は、**必ず以下のテスト用データを用いてグリッドを描画させる指示**を組み込んでください。

**テストデータ・パス:**
`C:\Users\kazuh\hoshutaro\src\data\equipments.json`

**対処法:**
1. テスト指示の最初のステップで、「このパスのファイルをアップロードボタンから読み込ませる」よう明確にサブエージェントに指示する。
2. または、ブラウザ実行前にメインAI側でテスト用JSONを `localStorage` に注入する JavaScript プロンプトをテスト指示に含める等して、**「データが描画されている状態」を作り出してから** UI機能の操作テストを進める。

## 2. browser_subagent への指示ガイドライン

`browser_subagent` を呼び出す際は、以下の点を含めて詳細かつ具体的なプロンプト（Task）を記述すること：

1. **初期アクセスURL**: どこにアクセスするか（例: `http://localhost:5174/`）。
2. **目的と焦点**: 今回の検証の主要目的（例: 「新設したヘッダーのボタンを目視したい」「ダイアログが開くかをテストしたい」）。
3. **具体的な操作手順**:
   - ✖ 「UIが変わったか確認して」
   - 〇 「1. 右下の設定ボタンをクリックする。 2. 開いたドロップダウンメニューの中に『アップロード』が表示されているか確認する。」
4. **確認と記録（必須）**: 
   - アクションの成否がどうであれ、必ず「スクリーンショットを撮影する」または「結果をテキストで報告する」ように要求する。

## 3. テストのステップとチェックリスト

エージェント自身で以下のステップを踏んでテストを完了させます：

1. **アクセスと描画チェック**: 指定URLにアクセス後、画面がロードできているか（白画面エラーになっていないか）を確認する。
2. **インタラクションテスト**: ユーザーの代わりに対象箇所の「クリック」「ホバー」等を実行させ、予期したピクセル変動やダイアログのPop-upが起こるかを確認する。
3. **視覚確認**: サブエージェントの結果として上がってきたスクリーンショット（Artifactsやログ）を見て、デザインが崩れていないか、アイコンが正常か、指示通りのレイアウトになっているか、Antigravity自身が（ユーザーに任せず）画像分析して結論を出す。

## 4. エラー時の自己修正（Self-Correction）

- もし `browser_subagent` がエラー（要素が見つからない、クリックできない）を返した場合、直ちにコードを修正すること。
- エラーの原因がサーバー停止の場合は、ユーザーにサーバー再起動を依頼するか、ターミナルコマンドで立ち上げる。
