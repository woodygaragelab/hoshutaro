# Frontend Editing Safety Protocol

このスキルは、**コード（特に `.ts`, `.tsx`, `.jsx` などのフロントエンドファイル）を編集した直後に、アプリケーションをクラッシュ（Unexpected Application Error 等）させないための必須プロトコル**です。
AI（Antigravity）は、ファイルの修正・置換（`multi_replace_file_content` の使用等）を行った後、**ユーザーに報告する前に必ずこの手順を実行**しなければなりません。

## なぜこのチェックが必要なのか？
AIによるコード置換では、閉じ括弧 `}` `)` の対応漏れや、インデントずれによる不要なトークンの混入など、意図しないSyntax Error（構文エラー）が発生することがあります。このまま放置してユーザーに画面確認を依頼すると、Vite開発サーバーがクラッシュし、ブラウザ上にエラー画面が表示されて業務がストップしてしまいます。これを防ぐために、AI自身が速やかに構文チェックとビルドチェックをパスする必要があります。

## 必須実行ステップ（自己修復ループ）

コード編集後、以下の順番でターミナルコマンドを実行し、エラーが出ないことを確認してください。

### ステップ 1: 高速構文チェック (esbuild)
置換直後、一番最初に編集したファイルに対して `esbuild` を使った高速な構文解析を行います。これにより、括弧の数や単純な文法エラーを0.1秒で即座に発見できます。

**実行コマンド例:**
```powershell
npx esbuild src/components/Path/To/EditedFile.tsx
```

*※出力が正常に通り抜ければ（Exit Code: 0）、基本的な括弧の対応やインデントミスによる構文崩壊は起きていません。*
*※もし `[ERROR] Unexpected ")"`などのエラーが出た場合は、対象行を `view_file` で確認し、修正後に再度このコマンドを実行してください。*

### ステップ 2: 型チェック (TypeScript)
構文が正しくても、「関数の引数が合っていない」「変数が未定義（ReferenceError）」といった型の不整合があると、実行時エラーや動作不良を引き起こします。

**実行コマンド例:**
```powershell
npx tsc --noEmit
```

*※エラーが出た場合、型定義やインポートの漏れがないか確認し、自己修正してください。*

### ステップ 3: 全体ビルドチェック (Vite Build)
局所的な構文や型だけでなく、Vite モジュールバンドラーが全体を結合（Import/Export）する際にコンフリクトやモジュール解決エラーが起きないか確認します。これを行うことで、Vite 開発サーバー (HMR) が「Failed to fetch dynamically imported module」のようなクラッシュを起こすことを完全に予防できます。

**実行コマンド例:**
```powershell
npx vite build
```

*(またはエラーメッセージのみを抽出する形で実行)*
```powershell
npx vite build > build.log 2>&1 ; type build.log
```

## エラーが発見された場合のルール
- 決して、「エラーが起きたままの状態でユーザーに『確認してください』と伝える」ような行動をとらないでください。
- ターミナルから返ってきたエラー内容（Line番号など）を基に、エラーがゼロになるまで `multi_replace_file_content` などで修正を繰り返し（Self-Correction）、全てパスした状態になってからユーザーに報告してください。
