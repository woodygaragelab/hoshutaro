# HOSHUTARO 会話型AIエージェント改善プラン

## Context

現在のHOSHUTAROのAIチャットは、キーワードベースのルーティング（"計画"→SchedulePlanner, "Excel"→ExcelImport等）で機械的に処理を振り分けている。そのため、ユーザーが自然に会話しても「保全計画作成に必要な情報を提示しました」のような機械的な応答しか返せない。

**目的**: LLMベースのインテント分類と会話ファーストの設計に切り替え、ユーザーとの自然なキャッチボールを通じてデータ操作ができるようにする。

---

## 現在のプロセスフロー（問題点付き）

```
ユーザーメッセージ
  ↓
┌─────────────────────────────────┐
│  orchestrator.route_and_execute │  ← キーワードマッチ("計画","Excel"等)
│  skill_loader.match_skill()    │  ← パターンマッチで分岐
└──────┬──────┬──────┬───────────┘
       ↓      ↓      ↓
  ExcelImport  Schedule  GenericChat/EditAgent
  (即実行)     (即計算)  (最小限のsystem prompt)
       ↓      ↓      ↓
  機械的な完了報告を chat_stream / pure_chat_stream でストリーム
```

**問題点:**
1. キーワードが含まれるだけで即座にエージェント実行 → 確認なし
2. 会話履歴を見ていない（毎回フロントから送り直し、サーバーで保持しない）
3. GenericChatのsystem promptが貧弱（「親切に答えてください」のみ）
4. データを参照した自然な会話ができない
5. 操作完了後の報告が別のLLM呼び出しで生成 → 文脈断絶

---

## 新しいプロセスフロー（並列LLM呼び出し）

### 設計方針: 全フローにおける並列/直列LLM呼び出しの戦略

全フローで発生するLLM呼び出し7箇所を洗い出し、
**どこを並列化すべきか、どこは依存関係で直列にせざるを得ないか**を明確にする。

#### LLM呼び出し一覧と並列化判定

```
┌─────┬──────────────────────────┬──────────┬───────────────────────────────────┐
│ No. │ LLM呼び出し               │ 並列/直列 │ 理由                              │
├─────┼──────────────────────────┼──────────┼───────────────────────────────────┤
│  1  │ classify_intent (分類)    │ ⚡並列    │ 2の会話ストリームと独立して実行可能 │
│  2  │ conversational_stream    │ ⚡並列    │ 1の分類と独立。ユーザー応答を先行   │
│     │ (会話応答)                │          │                                   │
├─────┼──────────────────────────┼──────────┼───────────────────────────────────┤

├─────┼──────────────────────────┼──────────┼───────────────────────────────────┤
│  4  │ Planner.predict          │ 🔗直列   │ 3の抽出結果+analyze統計に依存      │

│  6  │ ExcelImport.parse        │ 🔗直列   │ ファイルバイナリ解析結果に依存      │
│     │ (マッピング推論)         │          │ analyze_excel_structure()の後      │
├─────┼──────────────────────────┼──────────┼───────────────────────────────────┤
│  7  │ completion_report        │ 🔗直列   │ エージェント実行結果に依存          │
│     │ (完了報告生成)           │          │ 操作完了後にしか生成できない        │
```

```
  ┌─────────────────────────────────────────┐
  │  ┌──── 🔀 条件付き並列 ────┐            │
  │  │ 2が完了前に1が完了している│            │
  │  │ → 1の結果(params)で     │            │
  │  │   3.extract を先行起動可能│            │
  │  │   (2のストリーム中に)    │            │
  │  └──────────┬──────────────┘            │
  │             ↓                           │

```

**パターン3: データ編集（data_editing）**
```
  ┌────────⚡ 並列 ⚡────────────────────────┐
  │ 1.classify  2.conv_stream               │
  │             ↕ 2のストリーム中に...        │
  │             5.reasoning を先行起動可能    │
  │             (1の結果confirmedかつ         │
  │              confidence≧0.7の場合)       │
  └─────┬──────────────────────┬─────────────┘
        ↓                      ↓
  2をユーザーに送信       5のJSON操作生成が完了
                                ↓ 🔗直列(JSON結果に依存)
                          tools実行
                                ↓ 🔗直列
                          7.completion_report
  LLM呼び出し: 最大4回（1+2が並列、5は1完了後に起動、7は直列）
```

**パターン3 最適化: 3並列起動**
```
  ┌──────────────⚡ 3並列 ⚡──────────────────┐
  │ 1.classify  2.conv_stream  5.reasoning   │
  │ (JSON)      (ストリーム)    (JSON)         │
  │ temp=0.1    temp=0.7       temp=0.1       │
  │ tok=256     tok=1024       tok=1024       │
  └───┬─────────────┬──────────────┬──────────┘
      ↓             ↓              ↓
  分類結果      ユーザーに送信   JSON操作結果
      ↓                            ↓
  confidence確認               [confidence≧0.7?]
  ┌────────────────────────────────┤
  │ Yes: 5の結果を使ってtools実行  │
  │ No:  5の結果を破棄            │
  └────────────────────────────────┘

  ※ 5のreasoningは投機的に実行。confidenceが低ければ結果を捨てる。
  ※ トレードオフ: LLMコスト(無駄な呼び出し) vs レイテンシ(並列で高速化)
  → ローカルLLM(Ollama等)なら投機実行のコストは低い → 採用
  → クラウドAPI(GPT-4等)なら投機実行のコストが高い → 採用しない
  → adapter.supports_speculative で切り替え可能にする
```

**パターン4: Excel取込み**
```
  ファイルアップロード
        ↓ 🔗直列(ファイル解析が先)
  analyze_excel_structure() (LLM不要)
        ↓ 🔗直列(プレビューデータに依存)
  6.infer_mapping_with_llm(マッピング推論)
        ↓ 🔗直列(マッピング結果に依存)
  ユーザー確認待ち
        ↓ 🔗直列(確認後に処理開始)
  チャンク分割処理 (LLM不要、決定論的)
  LLM呼び出し: 1回のみ（全て直列、並列化の余地なし）
```

#### 実装上の並列化メカニズム

```python
# chat.py: event_generator() での並列起動パターン

# === パターン2/3: 分類 + 会話 + エージェント前処理を3並列 ===
async def event_generator():
    # 全LLM呼び出しを可能な限り早く起動
    intent_task = asyncio.create_task(
        adapter.classify_intent(messages, classifier_prompt)        # Call A
    )
    conv_gen = adapter.conversational_stream(messages, rich_prompt) # Call B (lazy)

    # Call Bのストリームを即座にユーザーへ送信開始
    full_response = ""
    async for chunk in conv_gen:
        full_response += chunk
        yield text_delta(chunk)

        # === 🔀 条件付き並列: Call Aが先に完了していればエージェント前処理を開始 ===
        if not intent_task.done():
            continue  # まだ分類結果が出ていない
        if agent_task is None:
            intent_result = intent_task.result()
            if intent_result["confidence"] >= 0.7:
                # Call Bストリーム中にエージェントを先行起動
                agent_task = asyncio.create_task(
                    execute_agent(intent_result, session_id, context)
                )

    session.append_message("assistant", full_response)

    # Call Bストリーム完了。Call Aの結果を確認。
    intent_result = await intent_task
    if intent_result["intent"] == "converse" or intent_result["confidence"] < 0.7:
        yield [DONE]  # 会話のみ
        return

    # エージェントが先行起動済みならawait、未起動なら今起動
    if agent_task is None:
        agent_task = asyncio.create_task(
            execute_agent(intent_result, session_id, context)
        )
    agent_result = await agent_task

    # 完了報告ストリーム（エージェント結果に依存 → 直列）
    if agent_result.get("operations"):
        async for chunk in adapter.completion_report_stream(agent_result["final_response"]):
            yield text_delta(chunk)
        yield data_model_update(session.data_model)

    yield [DONE]
```

**まとめ: 並列化で得られる効果**
```
直列実行の場合:  分類(500ms) → 会話(2000ms) → エージェント(1500ms) → 報告(800ms) = 4800ms
並列実行の場合:  [分類 + 会話](2000ms) → [エージェント※先行起動](500ms) → 報告(800ms) = 3300ms
                 ※ Call Bストリーム中にエージェントが先行開始しているため、待ち時間が短縮
```

## エージェント別 ReActループ処理フロー

### 全体のReActアーキテクチャ

```
┌─────────────────────────────────────────────────────────────────────┐
│                     統合 ReAct ループ                                │
│                                                                     │
│  ┌─────────┐    ┌─────────┐    ┌──────────┐    ┌────────────────┐  │
│  │ Reason  │───→│  Act    │───→│ Observe  │───→│  Evaluate      │  │
│  │ (LLM)   │    │ (Tool)  │    │ (結果)   │    │  (成功/失敗/   │  │
│  │         │    │         │    │          │    │   情報不足)     │  │
│  └─────────┘    └─────────┘    └──────────┘    └───────┬────────┘  │
│       ↑                                                │           │
│       │           ┌───────────┐                        │           │
│       └───────────│ 再推論    │←── 失敗/情報不足 ──────┘           │
│                   └───────────┘                        │           │
│                                              成功 ─────┘           │
│                                                ↓                   │
│                                      ┌─────────────────┐          │
│                                      │ Response (LLM)  │          │
│                                      │ 自然言語で報告   │          │
│                                      └─────────────────┘          │
└─────────────────────────────────────────────────────────────────────┘
```

### ReAct共通ルール
- **最大ループ回数**: 3回（無限ループ防止）
- **失敗時**: エラー内容をReasonフェーズにフィードバックし再推論
- **情報不足時**: ユーザーへの質問を生成して会話に戻る（ループ脱出）
- **成功時**: 操作結果を自然言語で報告

### E. 統合フロー: 並列LLM呼び出し × ReActループ × レスポンスストリーム

以下は `chat.py` の `event_generator()` 内で実行される全体制御フローを示す。
**並列LLM呼び出し（Call A / Call B）の起動タイミング、ReActループへの遷移条件、
ストリームのマージ・切替え、各エージェントパスの分岐**を明確にする。

```
┌═══════════════════════════════════════════════════════════════════════════┐
║  chat.py: event_generator() — 全体制御                                    ║
╞═══════════════════════════════════════════════════════════════════════════╡
│                                                                           │
│  ──── 前処理 ────                                                         │
│  1. session.append_message("user", user_msg)                              │
│  2. data_summary = format_conversational_context(session.data_model)       │
│  3. history = session.get_recent_history(10)                              │
│  4. adapter = get_llm_adapter(wait_timeout=2.0)                           │
│                                                                           │
│  ──── 特殊パス: Excelインポート待機中の場合 ────                           │
│  5. if session.import_state.status == "waiting_user":                     │
│       → Excel Phase 3 (チャンク分割処理) へ直行                           │
│       → ReActフロー C の STEP 5〜6 を実行                                │
│       → yield [DONE] & return                                            │
│                                                                           │
│  ──── 通常パス: 並列LLM呼び出し ────                                      │
│                                                                           │
│  6. ┌──────── asyncio 並列起動 ──────────────────────────────┐            │
│     │                                                        │            │
│     │  Call A: classify_intent()     Call B: conv_stream()    │            │
│     │  ┌─────────────────────┐      ┌──────────────────────┐ │            │
│     │  │ 軽量インテント分類   │      │ リッチ会話ストリーム  │ │            │
│     │  │ system: 分類専用     │      │ system: HOSHUTARO     │ │            │
│     │  │   prompt             │      │   性格+ドメイン知識   │ │            │
│     │  │ + data_summary       │      │ + data_summary        │ │            │
│     │  │ + history            │      │ + history             │ │            │
│     │  │ temp=0.1             │      │ temp=0.7              │ │            │
│     │  │ max_tokens=256       │      │ max_tokens=1024       │ │            │
│     │  │ JSON応答(非stream)   │      │ ストリーム応答        │ │            │
│     │  └──────────┬──────────┘      └──────────┬───────────┘ │            │
│     │             ↓                             ↓              │            │
│     │  intent_task=create_task(A)    conv_gen = B (lazy gen)   │            │
│     └──────────────────────────────────────────────────────────┘            │
│                                                                           │
│  ════════ ストリームPhase: Call B → ユーザー ════════                      │
│                                                                           │
│  7. Call Bのストリームを即座にSSE送信開始                                   │
│     full_response = ""                                                    │
│     async for chunk in conv_gen:                                          │
│         full_response += chunk                                            │
│         yield {"type": "text_delta", "delta": chunk}                      │
│         # → ユーザーはt≒200msで最初の文字を見始める                       │
│                                                                           │
│  8. session.append_message("assistant", full_response)                    │
│                                                                           │
│  ════════ 判定Phase: Call A結果で分岐 ════════                             │
│                                                                           │
│  9. intent_result = await intent_task                                     │
│     # Call Aは軽量(256tokens)なのでPhase 7完了前に既に終わっているはず     │
│     # Phase 7完了後にawaitすることで、会話が必ず先にユーザーに届く         │
│                                                                           │
│  10. 分岐判定:                                                            │
│                                                                           │
│  ┌─── intent == "converse" OR confidence < 0.7 ───────────────────────┐   │
│  │                                                                     │   │
│  │  Phase 7で送信済みの会話応答がそのまま最終回答。                      │   │
│  │  エージェント実行なし。                                              │   │
│  │  yield [DONE]                                                       │   │
│  │                                                                     │   │
│  │  例: 「計画を立てたい」→ confidence=0.4 →                           │   │
│  │  Conv応答:「保全計画の作成ですね。現在14件の作業が登録されています。  │   │
│  │           どの機器の計画を作成しましょうか？」                        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                           │
│  ┌─── intent == "schedule_planning" AND confidence >= 0.7 ────────────┐   │
│  │                                                                     │   │
│  │  Conv応答はすでに送信済み（例:「P-101の履歴を分析しますね...」）      │   │
│  │                                                                     │   │
│  │  11-S. SchedulePlanner ReActループ実行:                              │   │
│  │     initial_state = {                                               │   │
│  │       session_id,                                                   │   │
│  │       messages: history + [{"role":"user","content": user_msg}],    │   │
│  │       data_context: session.data_model,                             │   │
│  │       target_asset_id: intent_result.params.target_asset_id,       │   │
│  │       user_specified_days: intent_result.params.specified_days     │   │
│  │     }                                                               │   │
│  │     final_state = await schedule_planner_engine.ainvoke(state)      │   │
│  │                                                                     │   │
│  │     ReActループ内部 (LangGraphグラフ):                               │   │
│  │     ┌─────────────────────────────────────────────────────────┐     │   │
│  │     │ extract─→[target?]─No→ return 質問テキスト             │     │   │
│  │     │              ↓ Yes                                      │     │   │
│  │     │         analyze─→[intervals>=2?]─No→ return 質問       │     │   │
│  │     │              ↓ Yes                                      │     │   │
│  │     │         predict─→propose─→[ops?]─No→ return 提案      │     │   │
│  │     │                               ↓ Yes                    │     │   │
│  │     │                          tools実行                      │     │   │
│  │     │                               ↓                        │     │   │
│  │     │                     [success?]─No→ retry (max 3)       │     │   │
│  │     │                          ↓ Yes                         │     │   │
│  │     │                     return (ops + final_response)       │     │   │
│  │     └─────────────────────────────────────────────────────────┘     │   │
│  │                                                                     │   │
│  │  12-S. 結果をストリーム送信:                                         │   │
│  │     if final_state.operations:                                      │   │
│  │       # 操作成功 → 完了報告をLLMストリームで生成                     │   │
│  │       async for chunk in adapter.completion_report_stream(          │   │
│  │           final_state.final_response, history):                     │   │
│  │         yield {"type": "text_delta", "delta": chunk}               │   │
│  │       yield {"type": "data_model_update", "data": session.data}    │   │
│  │     elif final_state.final_response:                                │   │
│  │       # 質問/提案テキスト → 擬似ストリームで送信                     │   │
│  │       for i in range(0, len(text), 2):                             │   │
│  │         yield {"type": "text_delta", "delta": text[i:i+2]}        │   │
│  │                                                                     │   │
│  │  yield [DONE]                                                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                           │
│  ┌─── intent == "data_editing" AND confidence >= 0.7 ─────────────────┐   │
│  │                                                                     │   │
│  │  11-E. EditAgent ReActループ実行:                                    │   │
│  │     final_state = await maintenance_engine.ainvoke(state)           │   │
│  │                                                                     │   │
│  │     ReActループ内部 (LangGraphグラフ):                               │   │
│  │     ┌─────────────────────────────────────────────────────────┐     │   │
│  │     │ routing → reasoning(LLM→JSON)                          │     │   │
│  │     │   ↓                                                     │     │   │
│  │     │ [JSON解析OK?]─No→ error feedbackで retry (max 3)       │     │   │
│  │     │   ↓ Yes                                                 │     │   │
│  │     │ [type=explain_only?]─Yes→ return テキスト               │     │   │
│  │     │   ↓ No                                                  │     │   │
│  │     │ tools実行 → return (ops + response)                     │     │   │
│  │     └─────────────────────────────────────────────────────────┘     │   │
│  │                                                                     │   │
│  │  12-E. 結果をストリーム送信: (12-Sと同じロジック)                     │   │
│  │  yield [DONE]                                                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
```

### 1. 新規作成: `backend/app/engine/conversational_dispatcher.py`

コアとなる新コンポーネント。**2つの並列LLM呼び出し**を管理する。

```python
class IntentResult(BaseModel):
    intent: Literal["converse", "schedule_planning", "data_editing", "excel_import"]
    parameters: dict = {}  # エージェントに渡すパラメータ（target_asset_id等）
    confidence: float = 0.0

class DispatchResult(BaseModel):
    intent_result: IntentResult
    conversation_stream: AsyncGenerator[str, None]  # 会話ストリーム（Call B）

async def dispatch(
    session_id: str,
    user_message: str,
    data_context: dict,
    conversation_history: list[dict],
    adapter: LLMAdapter
) -> DispatchResult:
    # 1. データコンテキストを人間可読サマリーに変換
    context_summary = format_conversational_context(data_context)

    # 2. Call A: インテント分類（軽量JSON、temp=0.1, max_tokens=256）
    #    Call B: 会話ストリーム生成（リッチprompt、temp=0.7, ストリーミング）
    #    → asyncio.create_task で Call A を並列起動
    intent_task = asyncio.create_task(
        adapter.classify_intent(messages, classifier_prompt)
    )
    conv_stream = adapter.conversational_stream(messages, rich_prompt)

    # 3. Call Aの結果を待つ（軽量なので先に返る）
    intent_result = await intent_task
    # 4. confidence < 0.7 → intent を "converse" に上書き

    return DispatchResult(intent_result=intent_result, conversation_stream=conv_stream)
```

```python
async def event_generator():
    session.append_message("user", user_msg)

    result = await route_and_execute(...)
    intent = result["intent_result"]
    conv_stream = result["conversation_stream"]

    # Step 1: Call Bの会話ストリームを即座にユーザーへ送信開始
    full_response = ""
    async for chunk in conv_stream:
        full_response += chunk
        yield {"data": json.dumps({"type": "text_delta", "delta": chunk})}

    session.append_message("assistant", full_response)

    # Step 2: 操作が必要な場合はエージェント実行
    if intent["intent"] != "converse" and intent["confidence"] >= 0.7:
        agent_result = await execute_agent(
            intent["intent"], intent.get("parameters", {}), session_id, context
        )
        ops = agent_result.get("operations", [])
        if ops:
            # 操作完了報告を追加ストリーム
            summary = agent_result.get("final_response", "")
            async for chunk in adapter.completion_report_stream(summary):
                yield {"data": json.dumps({"type": "text_delta", "delta": chunk})}
            yield {"data": json.dumps({"type": "data_model_update", "data": session.data_model})}

    yield {"data": "[DONE]"}
```
